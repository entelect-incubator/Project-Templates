# Kestrel Port Configuration - Master Summary

## Your Questions - Answered

### Q1: Does hardcoded port need to match launchSettings.json?
**NO.** They are independent:
- launchSettings.json: Used by `dotnet run` debugging only
- Program.cs Config: Used in ALL environments (dev, prod, docker, k8s)

### Q2: Can the port come from configuration instead of hardcoding?
**YES!** Read from `appsettings.json` using `IConfiguration`

### Q3: What's the better way?
**Use environment-based configuration:**
- Base config in appsettings.json
- Environment-specific in appsettings.Development.json, appsettings.Production.json
- Runtime overrides via environment variables

## ✅ Solution (One-Minute Version)

### Change Program.cs from:
```csharp
builder.WebHost.ConfigureKestrel(options =>
{
    options.ListenAnyIP(7160, listenOptions => 
        listenOptions.Protocols = HttpProtocols.Http1AndHttp2AndHttp3);
});
```

### To:
```csharp
builder.WebHost.ConfigureKestrel((context, options) =>
{
    var port = int.Parse(context.Configuration["Kestrel:Port"] ?? "5000");
    
    options.ListenAnyIP(port, listenOptions =>
    {
        listenOptions.Protocols = HttpProtocols.Http1AndHttp2AndHttp3;
    });
});
```

### Add to appsettings.json:
```json
{
  "Kestrel": {
    "Port": 5000
  }
}
```

### That's it! Now:
```bash
dotnet run                          # Uses port 5000
Kestrel__Port=7160 dotnet run       # Uses port 7160
docker run -e Kestrel__Port=8080    # Uses port 8080
```

## Key Concepts

### Port Configuration Priority
1. **Env Variable** (Highest) - `Kestrel__Port=8080`
2. **appsettings.{Environment}.json** - `appsettings.Production.json`
3. **appsettings.json** - Base config
4. **Code Default** - `?? "5000"`
5. **ASP.NET Default** (Lowest) - Port 5000

### Why This Matters
✅ DevOps best practice  
✅ Docker/Kubernetes ready  
✅ CI/CD friendly  
✅ Team consistency  
✅ No hardcoding  
✅ Easy to maintain  

### When to Use Each

| Scenario | Solution |
|----------|----------|
| Local dev | Use appsettings.json (port 5000) |
| Test different port | `Kestrel__Port=X dotnet run` |
| Docker container | `-e Kestrel__Port=5000` |
| Production | appsettings.Production.json |
| CI/CD pipeline | Environment variables |

## Real-World Examples

### Example 1: Basic Local Development
```bash
$ dotnet run
# Uses: appsettings.Development.json → port 5000
```

### Example 2: Override for Testing
```bash
$ Kestrel__Port=7160 dotnet run
# Uses: port 7160 (env var wins)
```

### Example 3: Docker Development
```bash
$ docker build -t myapp .
$ docker run -p 5000:5000 -e Kestrel__Port=5000 myapp
# Container listens on 5000, accessible on host port 5000
```

### Example 4: Docker Production
```bash
$ docker run -e ASPNETCORE_ENVIRONMENT=Production myapp
# Uses: appsettings.Production.json → port 80
```

### Example 5: Kubernetes
```yaml
containers:
- name: api
  image: myapp:latest
  ports:
  - containerPort: 5000
  env:
  - name: Kestrel__Port
    value: "5000"
  - name: ASPNETCORE_ENVIRONMENT
    value: "Production"
```

## FAQ

**Q: Does this break anything?**  
A: No. Fully backward compatible. Old code still works.

**Q: Do I have to do this?**  
A: No, but it's a best practice. Strongly recommended for production apps.

**Q: Why not use Kestrel's built-in JSON config?**  
A: That approach works too, but this is more explicit and flexible.

**Q: What if I forget the double underscore in Kestrel__Port?**  
A: Use `Kestrel__Port` (double underscore) or `Kestrel:Port` in JSON. Never `Kestrel_Port` (single underscore won't work).

**Q: Which projects need this?**  
A: All 5 template projects for consistency.
